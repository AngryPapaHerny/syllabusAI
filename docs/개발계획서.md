## 1. 개요

| 항목 | 내용 |
|---|---|
| 제품 | 입력(주제·수준·분량) 기반 개인 맞춤 커리큘럼/콘텐츠 자동 생성 + 평가·재학습 플랫폼 |
| 1차 타깃 | 코딩/IT 학습자 (이후 일반 도메인 확장) |
| 런타임 | 웹앱(Next.js) + Supabase(Postgres + pgvector) |
| LLM | 프로바이더 무종속(LLM Gateway): Claude·GPT·Gemini 교체·혼용 |
| 콘텐츠 포맷 | P-C-S-M-A 유닛 |
| 개발 형태 | 소규모 팀(1~2명) + AI 코딩 보조 가정 |

### 개발 목표 (우선순위)
1. **학습 루프 증명** — 생성→학습→평가→재학습이 실제로 작동(Phase 0).
2. **재사용 자산화** — 개념 라이브러리로 한계비용 절감(Phase 1).
3. **개인화 정밀화** — 적응형 진단·지식 추적(Phase 2).
4. **도메인 확장·자가개선**(Phase 3).

## 2. 범위 (In / Out)

### Phase 0 (MVP) In
- 코딩 단일 도메인, 단일 주제 템플릿 흐름
- per-user 생성(재사용 라이브러리 **없음**) — 콜드 스타트 회피
- 온보딩 → (간이)진단 → 커리큘럼 생성 → 학습 뷰 → 평가 → 재학습(다른 변형) → 진도
- LLM Gateway 최소 구현(2개 프로바이더 이상)
- 코드 실행 샌드박스(예제 검증 + 실습 채점)

### Phase 0 Out (의도적 보류)
- pgvector 의미 검색 기반 재사용/조립 → Phase 1
- IRT 적응형 진단, 지식 추적, 간격 반복 → Phase 2
- 일반 도메인, 커뮤니티, 콘텐츠 자동개선 루프 → Phase 3
- 결제/구독, 팀/기관 기능

## 4. 데이터 모델 (Phase 0 적용 스키마)

설계도 11개 테이블 중 Phase 0에 필요한 핵심만. (벡터 컬럼은 Phase 1에 추가)

```sql
-- 사용자 프로필 (Supabase auth.users 확장)
profiles(user_id pk→auth.users, display_name, created_at)

-- 커리큘럼
curricula(id pk, owner_id→profiles, goal_text, domain, level_target,
          time_budget_hours_per_week, status, created_at)

-- 커리큘럼 구성 유닛
curriculum_units(id pk, curriculum_id→curricula, concept_key text, title,
                 order_idx int, role text /* core|optional|remediation */, status)

-- 유닛 콘텐츠 변형 (P-C-S-M-A)
unit_variants(id pk, concept_key text, level text, format text /* analogy|code|visual */,
              content jsonb /* {P,C,S,M,A} */, source_meta jsonb, quality_score numeric,
              status text, created_at)

-- 평가 문항 은행
assessment_items(id pk, concept_key text, type text, stem text, options jsonb,
                 answer jsonb, rationale text, difficulty numeric)

-- 학습자 숙련도 (지식 추적의 단일 진실원)
learner_concept_mastery(user_id, concept_key, mastery numeric /*0~1*/,
                        last_seen timestamptz, next_review_at timestamptz, attempts int,
                        pk(user_id, concept_key))

-- 시도 로그 (원인별 진단)
attempts(id pk, user_id, item_id→assessment_items, answer jsonb, correct bool,
         failure_type text /* gap|misconception|slip|null */, created_at)

-- 생성 작업 큐
gen_jobs(id pk, type text, payload jsonb, status text /* queued|running|done|failed */,
         priority int, provider text, model text, tokens int, cost numeric,
         created_at, finished_at)
```

- **RLS**: `curricula`/`learner_concept_mastery`/`attempts`는 `owner_id = auth.uid()` 기준 행 보안.
- **인덱스**: `concept_key`(조립·재사용 준비), `gen_jobs(status, priority)`, `next_review_at`.
- Phase 1에서 `concepts`(+`embedding vector`), `concept_prerequisites`, `content_flags` 추가.

## 5. 생성 파이프라인 (Phase 0 구현 수준)

설계도의 4단계를 MVP에서는 **단순화**해 구현하되 인터페이스는 확장 가능하게.

| 단계 | Phase 0 구현 | 확장(후속) |
|---|---|---|
| Calibrator | 주제→개념 목록 분해 + 수준 매핑(LLM 1콜) | 학습자 mastery 대비 인지부하 추정 |
| Grounding | 코드 예제 **실제 실행 검증** 우선 | 신뢰 소스 검색 + 출처 부착 |
| Writer | P-C-S-M-A를 JSON 스키마로 강제 출력 | 다중 변형 동시 생성 |
| Self-Check | 정답 테스트 통과 + 퀴즈 형식 검증 | 난이도 일관성·모호성 정밀 검사 |

- 모든 호출은 **LLM Gateway** 경유. 작업별 모델 정책(생성=상위, 검증=경량).
- 산출물은 `unit_variants.content`(jsonb)로 적재 → 런타임이 조립.
- 비동기: 온보딩 직후 `gen_jobs`에 적재, 워커가 처리, 런타임은 준비된 유닛부터 순차 노출.

## 6. 웹앱 화면 & API (Phase 0)

### 화면
1. **랜딩/로그인** (Supabase Auth)
2. **온보딩** — 주제·목표·수준·주당 시간 입력
3. **진단**(간이) — 개념별 자가/객관 문항 → mastery 초기화
4. **커리큘럼 대시보드** — 유닛 목록·진도바·생성 상태
5. **학습 뷰** — P-C-S-M-A 렌더 + 실습 에디터 + **AI 튜터(스트리밍)**
6. **평가 뷰** — 퀴즈/실습 채점 → 오답 시 재학습 분기
7. **진도/포트폴리오**

### 핵심 API (route handlers)
- `POST /api/curricula` — 온보딩 입력 → 커리큘럼 생성 + `gen_jobs` 적재
- `GET /api/curricula/:id` — 조립된 커리큘럼·유닛 상태
- `POST /api/diagnostic/submit` — 진단 결과 → mastery 초기화
- `POST /api/tutor` — 현재 유닛+learner model 컨텍스트로 **스트리밍** 응답
- `POST /api/assess/submit` — 답안 채점(객관/코드 실행) → `attempts` 기록 → 재학습 판정
- `POST /api/run` — 코드 샌드박스 실행(실습/검증 공용)
- `POST /api/gen/worker` — (내부) 큐 처리 엔드포인트/크론

## 7. 스프린트 계획 (2주 단위, 일정은 추정)

> 소규모 팀 + AI 보조 기준. 절대 일정보다 **순서·완료조건**에 의미.

| 스프린트 | 기간(추정) | 목표 | 핵심 산출물 |
|---|---|---|---|
| **S0 셋업** | ~2주 | 환경·스킬 검증 | 레포·CI·Supabase·LLM Gateway·샌드박스 PoC |
| **S1 데이터+생성코어** | ~2주 | 스키마 + 생성 파이프라인 백엔드 | 마이그레이션·`gen_jobs` 워커·유닛 1개 생성 |
| **S2 학습 뷰+튜터** | ~2주 | 콘텐츠 소비 경험 | 학습 뷰·P-C-S-M-A 렌더·스트리밍 튜터 |
| **S3 온보딩+진단** | ~2주 | 진입 흐름 | 온보딩·간이 진단·mastery 초기화·대시보드 |
| **S4 평가+재학습** | ~2주 | 루프 닫기 | 채점·`attempts`·재학습(다른 변형) 분기 |
| **S5 통합+검증** | ~2주 | MVP 완성 | E2E 시나리오·버그픽스·내부 시연 |

→ **MVP 목표: 약 10~12주.** (Phase 1 재사용 엔진은 별도 후속.)

## 8. 마일스톤 & 완료 조건(DoD)

- **M1 (S1 종료)**: 코딩 주제 1개로 P-C-S-M-A 유닛이 자동 생성되고, 코드 예제가 샌드박스에서 통과한다.
- **M2 (S2 종료)**: 생성된 유닛을 학습 뷰에서 보고, AI 튜터와 대화(스트리밍)할 수 있다.
- **M3 (S4 종료)**: 평가에서 일부러 오답 → **다른 변형**이 제공되고 `learner_concept_mastery`/`attempts`가 갱신된다.
- **M4 (S5 종료, = MVP)**: "파이썬 기초 알고리즘" 한 주제로 온보딩→진단→학습→평가→재학습 **end-to-end** 완주.

**일반 DoD**: 타입체크/린트 통과 · 핵심 경로 E2E 1개 · RLS로 타 사용자 데이터 격리 · LLM 호출 비용 로깅.

## 9. 테스트 & 검증 전략

- **단위**: LLM Gateway 어댑터(모킹), 채점 로직, mastery 갱신 규칙.
- **통합**: `gen_jobs` 워커가 유닛을 끝까지 생성·적재.
- **E2E**(Playwright 권장): 온보딩→학습→평가→재학습 시나리오.
- **콘텐츠 품질**: 생성 코드 예제 자동 실행 통과율, 퀴즈 형식 검증.
- **검증 시나리오**(설계도 §검증):
  1. 코딩 주제 1개 end-to-end 실행
  2. 오답 시 다른 변형 제공 + 상태 갱신 확인
  3. 코드 예제 실행 통과(Self-Check)
  4. (Phase 1) 동일 주제 2번째 사용자 → 조립 재사용률 측정

## 10. 후속 단계 개요 (계획만)

- **Phase 1 — 재사용 엔진**: `concepts`+임베딩, pgvector 의미 검색, 조립기(Assembler), 변형 캐싱, 재사용률 KPI.
- **Phase 2 — 적응·정밀화**: IRT 문항 은행, 적응형 진단, 지식 추적, 간격 반복, 원인별 재학습 고도화.
- **Phase 3 — 확장·자가개선**: 일반 도메인, `content_flags` 기반 자동 재생성, 커뮤니티·포트폴리오.
